# Типы предупреждений и трендов в Zoomos Check

## 1. One-liner

Система делит все проблемы выкачки на три слоя: **ERROR блокирует отправку**, **WARNING требует внимания**, а **TREND информирует об отклонениях** от исторической нормы.

---

## 2. Аналогия

Представь фабрику, выпускающую отчёты:

- **ERROR** — красный свет на конвейере, продукт брак, отправлять нельзя
- **WARNING** — жёлтый свет, продукт подозрительный, нужна дополнительная проверка, но может пройти
- **TREND** — информационное табло об историческом тренде, не останавливает конвейер, просто говорит "смотри, скорость снизилась на 30%"

**canDeliver** (можно ли отдавать клиенту) — это кнопка, которая загорается только если на конвейере нет красных огней (ERROR-ов).

## 4. Пошаговая логика

### Фаза 1: Анализ текущей выкачки (ERROR/WARNING)

**Файл:** `ZoomosCheckService.java:1382-1515`

```java
public GroupEvalResult evaluateAndBuildIssues(List<ZoomosParsingStats> sortedGroup,
                                               int dropThreshold,        // Default: 10%
                                               int errorGrowthThreshold, // Default: 30%
                                               ...)
```

**Логика присваивания критичности:**

1. **ERROR — падение "В наличии" на 100%** (строка 1453-1461)
   - Если был ненулевой stock (медиана или предыдущая выкачка), а теперь 0
   - Пример: `"В наличии: [медиана: 150] → 0 (−100%)"`
   - **Почему critical:** товар исчез с витрины, значит API сломался или база изменилась

2. **ERROR — падение "В наличии" > порога** (строка 1463-1469)
   - Если drop > dropThreshold (по умолчанию 10%)
   - Пример: `"Падение 'В наличии': [медиана: 150] → 50 (−66%)"`
   - **Почему critical:** резкое снижение может быть ошибкой парсера или реальной проблемой с API

3. **WARNING — 100% выкачка, но нет товаров** (строка 1410-1413)
   - Все записи в группе: `completionPercent >= 100` и `totalProducts == 0`
   - **Почему warning, не error:** фильтр может быть слишком узкий, нужна проверка конфига

4. **WARNING — первая запись, 0 товаров в наличии** (строка 1421-1426)
   - Одна-единственная запись в группе, и уже 0 in-stock
   - **Важность:** может быть нормально (новое место), но нужна проверка

5. **WARNING — выкачка незавершена** (строка 1434-1435)
   - `completionPercent < 100` — просто ставит флаг, без issue-сообщения
   - **Важность:** информационная, может завершиться через час

6. **WARNING — рост ошибок парсинга** (строка 1494-1507)
   - Если `newErrors >= minAbsoluteErrors` И `growth > errorGrowthThreshold`
   - Пример: `"Рост ошибок: 10 → 25 (+150%)"`
   - **Почему warning, не error:** парсер может восстановиться, нужно мониторить

### Фаза 2: Анализ трендов (TREND_WARNING)

**Файл:** `ZoomosCheckService.java:1129-1182` и `ZoomosAnalysisController.java:1033-1084`

**Когда срабатывает:**
- Если у run есть `baselineDays > 0` (по умолчанию 14 дней)
- Берёт медиану за предыдущие 14 дней
- Сравнивает с текущей выкачкой

**Три типа тренд-предупреждений:**

1. **TREND: Падение доли "В наличии"** (строка 1139-1148)
   - Если `stockRatio` упал на > dropThreshold (обычно 10%)
   - `curRatio = currentInStock / currentTotal`
   - `baselineRatio = median(inStock/total за 14 дней)`
   - Пример: было 85% → стало 60% (−29%)

2. **TREND: Рост ошибок** (строка 1152-1170)
   - По процентам от товаров: `errorRate = errorCount * 100 / totalProducts`
   - Если вырос относительно baseline на errorGrowthThreshold И абсолютная дельта > 0.5%
   - Или если **впервые появились ошибки:** `curRate > максимум(1%, errorThreshold * 10)`
   - Пример: было 2% ошибок → стало 4.5% (+125%)

3. **TREND: Замедление скорости выкачки** (строка 1173-1182)
   - `durationRate = parsingDurationMinutes * 1000 / totalProducts`
   - Если упал на > dropThreshold И абсолютная дельта > 0.1 мин/1000 товаров
   - Пример: было 0.45 мин/1000 → стало 0.72 мин/1000 (+60%)
   - **Почему это плохо:** замедление указывает на перегрузку или проблемы в сети

### Фаза 3: Вычисление вердикта (canDeliver)

**Файл:** `ZoomosAnalysisController.java:1028-1031`

```java
// NOTE: canDeliver вычисляется ДО добавления TREND_WARNING
boolean canDeliver = issues.stream()
        .noneMatch(i -> "ERROR".equals(i.get("type")));
```

**Правило:**
- ✅ **Отчёт готов — можно отдавать** = нет ERROR-ов в issues
- ❌ **Есть проблемы — проверьте** = хотя бы один ERROR присутствует
- WARNING не блокирует, только ERROR блокирует
- TREND_WARNING не блокирует — добавляется ПОСЛЕ вычисления canDeliver (строка 1080)

## 5. Ловушки

### Ловушка 1: порядок срабатывания и baseline

Если базовый период содержит планомерно снижающиеся данные, медиана может не отразить текущее "восстановление" как позитивное. Тренды используют отдельные пороги `trendDropThreshold` и `trendErrorThreshold` (строка 1068-1069) и смотрят на **медиану**, не на отдельные дни.

### Ловушка 2: что такое "медиана"

**Файл:** `ZoomosCheckService.java:1365-1370`

```java
private static Integer median(List<Integer> sorted) {
    if (sorted.isEmpty()) return null;
    int n = sorted.size();
    if (n % 2 == 0) return (sorted.get(n / 2 - 1) + sorted.get(n / 2)) / 2;
    return sorted.get(n / 2);
}
```

Медиана вычисляется на **отсортированном списке**, поэтому выбросы (случайный пик) почти не влияют. Но если выбросов много, медиана может быть неточной. Один случайный пик (например, баг парсера дал 150 вместо 10) не исказит базовую линию.

### Ловушка 3: ignoreStock флаг

**Файл:** `ZoomosCheckService.java:1444`

```java
if (!ignoreStock && !alwaysZeroInStock) {
    // PRIMARY: анализируем inStock
} else {
    // FALLBACK: анализируем totalProducts вместо inStock
}
```

Если `ignoreStock = true` (сайт не предоставляет информацию о наличии):
- Пропускаются все проверки inStock
- Вместо этого проверяются totalProducts (падение товаров)
- Если товары тоже 0, выдаётся WARNING "Всегда 0 в наличии" (строка 1510-1516)

**Ловушка:** если забыть установить `ignoreStock`, будут ложные ошибки о нулевом наличии, хотя сайт их просто не предоставляет.

## 6. Блок "Тренды" в UI

### Где находится

**HTML:** `check-results.html:452-470` — БЛОК 2.5, свёрнут по умолчанию, синяя полоса (info-color)

```html
<div th:if="${trendIssues != null and !trendIssues.isEmpty()}" class="card mb-3 border-info">
    <div class="card-header bg-info bg-opacity-10">
        <i class="fas fa-chart-line me-1 text-info"></i>Тренды
        <span class="badge bg-info text-dark">${trendIssues.size()}</span>
    </div>
```

### Как вычисляется

1. `baselineDays` (обычно 14) дней ДО даты `dateFrom` выкачки
2. Загружаются все выкачки за этот период (строка 1033-1037)
3. Вычисляется медиана по трём метрикам (строка 1065):
   - `stockRatio` = медиана(inStock / totalProducts)
   - `errorRate` = медиана(errorCount / totalProducts * 100)
   - `durationRate` = медиана(parsingDurationMinutes * 1000 / totalProducts)
4. Текущая выкачка сравнивается с baseline (строка 1129-1182)

### Три типа тренд-сообщений

| Тип | Сообщение | Что означает | Пример |
|-----|-----------|-------------|--------|
| **Stock Ratio** | `Доля 'В наличии' снизилась: было X% → стало Y% (−Z%)` | Товаров в наличии меньше, чем обычно | было 80% → стало 50% |
| **Error Rate** | `Рост ошибок: было X% → стало Y%` или `Появились ошибки: X%` | Парсер ошибается чаще, чем раньше | было 2% → стало 5% |
| **Duration Rate** | `Скорость снизилась: было X.XX → стало Y.YY мин/1000 тов.` | Выкачка идёт медленнее | было 0.4 → стало 0.7 мин/1000 |

### Почему тренды НЕ блокируют отправку

Это **исторический контекст**, не текущая проблема:
- TREND ≠ проблема в выкачке (это анализ за две недели)
- TREND = предупреждение об изменении (что-то стало по-другому)

Позволяет видеть **грядущие проблемы** до того, как они станут критичными.

## Итоговая таблица критичности

| Тип | Цвет | Блокирует Deliver | Важность | Действие |
|-----|------|------------------|----------|----------|
| **ERROR** | Красный | **Да** | Критична | Разбираться — проблема в выкачке сейчас |
| **WARNING** | Жёлтый | Нет | Высокая | Проверить, может быть временное |
| **TREND** | Синий | Нет | Средняя | Мониторить — может указывать на проблему завтра |

**Главное правило:** если нет ERROR-ов → `canDeliver = true` → можно отдавать клиенту.

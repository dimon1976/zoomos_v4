# Процесс работы по рефакторингу

## Цель

Пошаговые правила для code-ассистента по выполнению рефакторинга согласно `doc/tasklist_refactor.md`.

**Базовые документы:**
- `@vision_refactor.md` - техническое видение
- `@conventions_refactor.md` - соглашения по коду  
- `doc/tasklist_refactor.md` - план работы

---

## 🔄 Процесс выполнения итерации

### 1. Подготовка (ОБЯЗАТЕЛЬНО)
- [ ] Прочитать задачи текущей итерации из `doc/tasklist_refactor.md`
- [ ] Изучить цель итерации и критерии готовности
- [ ] Убедиться что предыдущая итерация завершена

### 2. Планирование решения
- [ ] **СОГЛАСОВАТЬ** предполагаемое решение с пользователем
- [ ] Показать **конкретные отрезки кода** которые будут изменены
- [ ] Описать архитектурные изменения
- [ ] Получить **ОДОБРЕНИЕ** перед началом кодирования

**❌ НЕ начинать кодирование без согласования!**

### 3. Реализация  
- [ ] Строго следовать `@conventions_refactor.md`
- [ ] Сохранять обратную совместимость
- [ ] Добавлять логгирование (@Slf4j)
- [ ] Использовать валидацию где требуется
- [ ] Обрабатывать исключения централизованно

### 4. Тестирование
- [ ] Запустить приложение: `mvn spring-boot:run -Dspring-boot.run.profiles=silent`
- [ ] Проверить основную функциональность
- [ ] Выполнить тест из текущей итерации
- [ ] Убедиться что нет регрессий

### 5. Завершение итерации
- [ ] Обновить прогресс в `doc/tasklist_refactor.md`
- [ ] Отметить выполненные задачи (✅)
- [ ] Обновить статус в таблице прогресса
- [ ] **ОЖИДАТЬ подтверждения** от пользователя
- [ ] Сделать коммит после одобрения

### 6. Переход к следующей итерации
- [ ] **СОГЛАСОВАТЬ** переход к следующей итерации
- [ ] Не переходить автоматически без одобрения

## 📋 Обязательные проверки

### Перед кодированием
- ✅ Решение согласовано с пользователем
- ✅ Показаны конкретные изменения кода
- ✅ Получено одобрение на реализацию

### После реализации  
- ✅ Приложение компилируется без ошибок
- ✅ Приложение запускается и работает
- ✅ Выполнены тесты из итерации
- ✅ Обновлен `doc/tasklist_refactor.md`

### Перед коммитом
- ✅ Пользователь подтвердил готовность итерации
- ✅ Все задачи итерации отмечены как выполненные
- ✅ Нет outstanding проблем

## 🚫 Запрещенные действия

### Архитектурные
- ❌ Изменять без согласования
- ❌ Удалять существующие классы/методы
- ❌ Переименовывать публичные интерфейсы
- ❌ Начинать следующую итерацию без одобрения

### Процедурные
- ❌ Кодировать без показа плана изменений
- ❌ Пропускать тестирование
- ❌ Коммитить без подтверждения
- ❌ Работать над несколькими итерациями одновременно

## 🎯 Принципы работы

### KISS (Keep It Simple)
- Простые решения предпочтительнее сложных
- Минимальные изменения для достижения цели
- Понятная коммуникация без избыточных деталей

### Безопасность изменений
- Каждое изменение тестируется
- Обратная совместимость сохраняется
- Возможность откатить изменения

### Прозрачность процесса
- Все изменения согласовываются заранее
- Прогресс отслеживается в реальном времени
- Пользователь контролирует каждый шаг

### Качество результата
- Код соответствует соглашениям
- Нет технических долгов
- Документация актуальна

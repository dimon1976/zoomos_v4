# Техническое видение рефакторинга Zoomos v4

## Цель и принципы
Создание максимально простого, поддерживаемого и расширяемого решения по принципу KISS (Keep It Simple, Stupid). Никакого оверинжиниринга - только самое необходимое.

---

## 1. Технологии

### Основной стек (сохраняем)
- **Java 17** - стабильная LTS версия
- **Spring Boot 3.2.12** - текущая версия проекта  
- **PostgreSQL** - надежная реляционная БД
- **Maven** - система сборки

### Дополнительные библиотеки (минимально необходимые)
- **Lombok** - упрощение кода (@Data, @Slf4j)
- **Apache POI 5.2.3** - работа с Excel
- **OpenCSV** - работа с CSV
- **Flyway** - миграции БД

### Frontend (упрощенный)
- **Thymeleaf** - серверные шаблоны
- **Bootstrap 5.3.0** - готовые компоненты
- **Vanilla JavaScript** - без фреймворков (KISS)
- **WebSocket/STOMP** - real-time уведомления

### Обработка ошибок и логирование
- **Единая система исключений** - GlobalExceptionHandler
- **Logback** - через Spring Boot с уровнями (ERROR, WARN, INFO, DEBUG)
- **Структурированное логирование** - одинаковый формат во всех модулях
- **Профили логирования** - silent/dev/verbose для разных сценариев

### Исключаем сложности
- ❌ Микросервисы - монолит проще  
- ❌ Docker/Kubernetes - пока не нужно
- ❌ Redis/кэш - преждевременная оптимизация
- ❌ Message queues - WebSocket достаточно
- ❌ Избыточное логирование - только необходимое

## 2. Принципы разработки

### Основные принципы
- **KISS** - простые решения вместо сложных архитектур, читаемый код важнее "умного"
- **DRY** - общие утилиты для повторяющихся операций, единые интерфейсы
- **SOLID (упрощенно)** - один класс = одна ответственность, расширяемость через интерфейсы
- **Fail Fast** - быстрая валидация, ранняя обработка ошибок, четкие сообщения

### Архитектурные подходы
- **Монолитная структура** - один JAR файл
- **Слоистая архитектура** - Controller → Service → Repository  
- **Декларативная конфигурация** - аннотации Spring
- **Минимум магии** - явные зависимости и конфигурации

### Качество кода
- **Единые стандарты** именования и форматирования
- **Минимум комментариев** - самодокументируемый код
- **Рефакторинг небольшими итерациями**
- **Единая обработка ошибок** во всех модулях

### Производительность (критично для больших файлов 400-500МБ)
- **Потоковая обработка** - не загружаем весь файл в память
- **Асинхронность** - фоновая обработка с прогресс-баром
- **Переиспользование объектов** - избегаем лишних allocation
- **Мониторинг памяти** - отслеживание потребления ресурсов

### Подход к разработке
- **Итеративность** - маленькими шагами с проверкой работоспособности
- **Тестирование потом** - сначала рефакторинг, затем покрытие тестами
- **Простота превыше всего** - если решение сложное, ищем проще

### Преимущества
- **Быстрый поиск** нужных сервисов по функциональности
- **Легкость расширения** - добавляем новые функции в соответствующие пакеты
- **Понятность структуры** для новых разработчиков
- **Минимум зависимостей** между пакетами

## 4. Архитектура проекта

### Архитектурные слои (простая 4-слойная архитектура)

```
┌─────────────────────────────────────┐
│         Presentation Layer          │ ← Контроллеры, WebSocket, Thymeleaf
├─────────────────────────────────────┤
│         Validation Layer            │ ← Валидация данных, файлов, бизнес-правил
├─────────────────────────────────────┤
│          Service Layer              │ ← Бизнес-логика, асинхронные операции
├─────────────────────────────────────┤
│           Data Layer                │ ← JPA Repositories, Entity, БД
└─────────────────────────────────────┘
```

### Слои в деталях

**1. Presentation Layer**
- REST API контроллеры + MVC контроллеры
- WebSocket endpoints для real-time уведомлений
- Thymeleaf шаблоны для UI
- DTO для входных/выходных данных

**2. Validation Layer (отдельно)**
- `FileValidationService` - проверка файлов (размер, формат, структура)
- `BusinessValidationService` - бизнес-правила и ограничения
- `TemplateValidationService` - валидация шаблонов импорта/экспорта
- `DataValidationService` - валидация данных в файлах

**3. Service Layer**
- Бизнес-сервисы (importing, exporting, core, utils)
- Асинхронные сервисы с выделенными ThreadPool
- Стратегии обработки (Strategy Pattern)
- Команды для асинхронных операций

**4. Data Layer**
- JPA Repositories
- Entity модели
- Flyway миграции

### Ключевые паттерны (простые и проверенные)

- **Strategy Pattern** - разные стратегии обработки (уже работает в редиректах)
- **Builder Pattern** - для сложных объектов (DTO, конфигурации)
- **Observer Pattern** - WebSocket уведомления о прогрессе
- **Template Method** - общая схема обработки файлов для импорта/экспорта

### Управление состоянием и производительность

**Stateless сервисы**
- Не хранят состояние между запросами
- Легко масштабируются и тестируются

**Асинхронные операции**
- Состояние операций сохраняется в БД
- WebSocket для real-time обновлений прогресса
- Выделенные ThreadPool для разных типов операций

**Обработка больших файлов (400-500МБ)**
- **Stream processing** - потоковое чтение без загрузки в память
- **Chunked processing** - обработка по частям (например, по 1000 строк)
- **Progress tracking** - сохранение прогресса в БД + WebSocket уведомления
- **Memory management** - контроль потребления памяти

### Принципы взаимодействия

- **Controller → Validation → Service → Repository**
- **Минимум связей** между слоями
- **Единая обработка ошибок** на всех уровнях
- **Fail Fast** - быстрая валидация на входе

## 5. Модель данных

### Принцип: Сохранить всё что работает

**Основное правило**: Все существующие сущности и их структура остаются без изменений или с минимальными модификациями для обеспечения совместимости.

### Существующие сущности (сохраняем как есть)

**Основные доменные сущности:**
- **Client** - клиенты системы (без изменений)
- **FileOperation** - операции с файлами (без изменений) 
- **ImportTemplate/ExportTemplate** - шаблоны конфигурации (без изменений)
- **Все остальные entity** - сохраняем существующую структуру

### Возможные дополнения (только при необходимости)

**Новые сущности только для новых функций:**
- **ValidationLog** - для детального логирования валидации (если потребуется)
- **SystemSettings** - для конфигурации системы (если ещё нет)
- **NotificationSettings** - настройки уведомлений (если потребуется)

### Подход к изменениям

**Разрешённые изменения:**
- ✅ Добавление новых полей (с default значениями)
- ✅ Добавление новых индексов для производительности  
- ✅ Создание новых таблиц для новой функциональности
- ✅ Flyway миграции для новых изменений

**Запрещённые изменения:**
- ❌ Удаление существующих полей
- ❌ Изменение типов данных существующих полей
- ❌ Переименование таблиц или полей
- ❌ Изменение существующих связей между таблицами

### Стратегия эволюции модели

1. **Фаза 1**: Анализ существующей модели данных
2. **Фаза 2**: Выявление узких мест без изменения структуры
3. **Фаза 3**: Добавление новых полей/таблиц только при крайней необходимости
4. **Фаза 4**: Оптимизация запросов и индексов

### Принципы работы с данными

- **Обратная совместимость** - новый код должен работать со старыми данными
- **Постепенная миграция** - поэтапное обновление без прерывания работы
- **Версионирование схемы** - через Flyway миграции
- **Rollback готовность** - возможность откатить изменения

### Заключение

Модель данных остаётся максимально стабильной. Все изменения только аддитивные (добавление) без breaking changes. Приоритет - сохранение работоспособности существующей функциональности.

## 6. Сценарии работы

### Основные пользовательские сценарии

#### 1. Система шаблонов (импорт/экспорт)

**Настройка шаблона импорта:**
- Создание/редактирование шаблона для клиента
- Настройка соответствий: заголовки файла → поля сущности
- Валидационные правила для входящих данных
- Переиспользование шаблонов между клиентами

**Настройка шаблона экспорта:**
- Создание/редактирование шаблона для клиента
- Настройка соответствий: поля сущности → заголовки файла  
- Дополнительные поля для статистики сравнения данных
- Переиспользование шаблонов между клиентами

#### 2. Импорт данных с шаблонами

**Процесс импорта:**
1. Выбор клиента и шаблона импорта
2. Загрузка файла Excel/CSV (до 500МБ)
3. Валидация файла по шаблону (структура, типы данных, бизнес-правила)
4. Асинхронная обработка с прогресс-баром и WebSocket уведомлениями
5. Детализированное логирование ошибок для последующего анализа
6. Уведомление о завершении или сбое операции

#### 3. Экспорт данных с шаблонами и стратегиями

**Процесс экспорта:**
1. Выбор клиента и шаблона экспорта
2. Выбор стратегии экспорта (правила обработки данных)
3. Применение бизнес-правил перед экспортом
4. Генерация файла по шаблону с дополнительной статистикой
5. Асинхронная генерация для больших объемов данных
6. Скачивание готового файла

**Стратегии экспорта:**
- Различные правила обработки и трансформации данных
- Фильтрация и группировка записей
- Форматирование и валидация выходных данных

#### 4. Утилиты (без сохранения в БД)

**HTTP редиректы:**
- Загрузка файла с URL для обработки
- Массовая обработка редиректов (стратегии: Curl → Playwright → HttpClient)
- Статистика обработки (количество успешных/неуспешных, время обработки)
- Экспорт результатов без сохранения исходных данных в БД

**Штрих-коды и другие утилиты:**
- Обработка и сопоставление штрих-кодов
- Статистика по операциям (количество найденных, процент совпадений)
- Результаты только для скачивания, без долгосрочного хранения

#### 5. Администрирование и мониторинг

**Управление:**
- CRUD операции для клиентов, шаблонов, стратегий
- Просмотр истории операций импорта/экспорта
- Анализ статистики использования системы

**Maintenance:**
- Автоматические задачи по очистке временных файлов
- Мониторинг производительности и использования ресурсов
- Архивация старых операций

#### 6. Централизованная обработка ошибок

**Логирование ошибок:**
- Единая система сохранения всех типов ошибок
- Детализация: тип операции, время, причина, стек-трейс, пользовательские данные
- Категоризация ошибок (валидация, бизнес-логика, системные сбои)
- Интерфейс для анализа ошибок и их частоты

**Типы ошибок:**
- Ошибки валидации файлов и данных
- Бизнес-правила нарушения
- Системные ошибки (превышение памяти, таймауты)
- Ошибки внешних сервисов (для утилит)

### Ключевые особенности сценариев

- **Асинхронность**: все длительные операции выполняются в фоне
- **Real-time обратная связь**: WebSocket уведомления о прогрессе
- **Переиспользование**: шаблоны и стратегии как переиспользуемые компоненты
- **Производительность**: обработка больших файлов (400-500МБ) без блокировки UI
- **Наблюдаемость**: детальное логирование для анализа и отладки

## 7. Деплой

### Структура файлов приложения

```
zoomos_v4/
├── target/

│   └── export/               # Экспортируемые файлы
```

### Управление приложением

**Запуск:**
- Ручной запуск через командную строку
- Остановка через Ctrl+C
- Просмотр логов в консоли

**Обновление:**
1. Остановить приложение
2. Собрать новую версию: `mvn clean package`
3. Запустить обновленное приложение

**Резервное копирование:**

- ❌ Автоматические обновления
- ❌ Мониторинг и алертинг (пока не нужны)

### Возможные улучшения (в будущем при необходимости)

## 8. Конфигурирование

### Принцип: "Настроил и забыл"

**Подход**: Единая система конфигурации через файлы properties с профилями для разных сценариев. Настройка один раз с долгосрочным использованием.

### Структура конфигурации

**Основные файлы:**
```
src/main/resources/
├── application.properties              # Базовая конфигурация
├── application-silent.properties       # Минимальное логирование (по умолчанию)
├── application-dev.properties          # Для разработки и отладки  
└── application-verbose.properties      # Детальное логирование и диагностика
```

**Профили использования:**
- **silent** - рабочий режим, минимум логов, быстрый запуск
- **dev** - разработка, умеренное логирование
- **verbose** - отладка проблем, максимальная детализация

### Ключевые области конфигурации

**База данных:**
```properties
spring.datasource.url=jdbc:postgresql://localhost:5432/zoomos_v4
spring.datasource.username=postgres
spring.jpa.hibernate.ddl-auto=none
```

**Файловая система:**
```properties
application.upload.dir=data/upload
application.export.dir=data/export
spring.servlet.multipart.max-file-size=1200MB
```

**Thread Pools (производительность):**
```properties
import.async.core-pool-size=1
import.async.max-pool-size=2
export.async.core-pool-size=2
export.async.max-pool-size=4
```

**Maintenance система:**
```properties
maintenance.scheduler.enabled=false  # По умолчанию выключена для безопасности
maintenance.scheduler.file-archive.cron=0 0 2 * * *
maintenance.scheduler.database-cleanup.cron=0 0 3 * * SUN
```

**Сервер и порт:**
```properties
server.port=8081
server.servlet.context-path=/
```

### Профили в деталях

**silent (рекомендуемый по умолчанию):**
- Логирование: ERROR и WARN только
- Быстрый запуск приложения
- Минимум диагностической информации
- Оптимизирован для производительности

**dev (для разработки):**
- Логирование: INFO уровень
- DevTools включен
- Умеренная диагностика
- Баланс между информативностью и производительностью

**verbose (для отладки проблем):**
- Логирование: DEBUG и TRACE
- SQL запросы в логах
- Детальная диагностика всех операций
- Максимальная информативность

### Управление конфигурацией

**Запуск с профилем:**
```bash
# По умолчанию (silent)
mvn spring-boot:run

# Конкретный профиль
mvn spring-boot:run -Dspring-boot.run.profiles=dev
mvn spring-boot:run -Dspring-boot.run.profiles=verbose
```

**Переопределение настроек:**
- Через переменные окружения при необходимости
- Через внешние application.properties файлы
- Через JVM параметры

### Принципы конфигурирования

- **Простота**: минимум файлов и параметров
- **Стабильность**: настроил один раз - работает долго
- **Безопасность**: потенциально опасные функции выключены по умолчанию
- **Производительность**: оптимизированные значения по умолчанию для работы с большими файлами
- **Обратная совместимость**: сохранение существующих настроек

## 9. Логгирование

### Принципы: Централизованное и структурированное логгирование

**Подход**: Единая система логгирования во всех модулях с уровневым управлением через профили. Исключение избыточного логирования - только необходимое.

### Централизованная архитектура логгирования

**Технический стек:**
- **@Slf4j** - аннотация Lombok во всех классах
- **Logback** - через Spring Boot (встроенная конфигурация)
- **Профили** - silent/dev/verbose для управления детализацией
- **Единый формат** сообщений во всех модулях

### Уровни логирования по профилям

**Silent (production):**
- **ERROR** - критические ошибки системы
- **WARN** - предупреждения о потенциальных проблемах
- Минимальный объем логов для производительности

**Dev (development):**
- **INFO** + ERROR/WARN - бизнес-события и операции
- Умеренная детализация для разработки
- Баланс информативности и производительности

**Verbose (debugging):**
- **DEBUG** + TRACE + INFO/WARN/ERROR - максимальная детализация
- SQL запросы, стек-трейсы, диагностика
- Для глубокого анализа и отладки проблем

### Что логируем (обязательно)

**Критические события:**
- Все исключения с полным контекстом
- Начало и завершение файловых операций (импорт/экспорт)
- Ошибки валидации файлов и данных
- Проблемы с внешними сервисами (утилиты)

**Бизнес-события:**
- Создание/изменение клиентов и шаблонов
- Запуск асинхронных операций
- WebSocket подключения и отключения
- Maintenance задачи и их результаты

**Производительность:**
- Время выполнения операций с большими файлами (>100МБ)
- Статистика обработки (количество записей, время выполнения)
- Превышение лимитов памяти или времени

### Чего НЕ логируем (принципы безопасности)

**Запрещено:**
- ❌ Персональные данные из файлов
- ❌ Пароли, токены, ключи API
- ❌ Полное содержимое больших файлов
- ❌ Дублирующиеся сообщения в циклах
- ❌ Избыточную техническую информацию

### Структура сообщений

**Единый формат:**
```java
// Пример централизованного логгирования
@Slf4j
@Service  
public class ImportService {
    
    public void processFile(String filename) {
        log.info("Starting file import: {}", filename);
        try {
            // business logic
            log.info("File import completed successfully: {}", filename);
        } catch (Exception e) {
            log.error("File import failed: {} - {}", filename, e.getMessage(), e);
        }
    }
}
```

### Централизованная обработка ошибок

**Единая система:**
- GlobalExceptionHandler логирует все неперехваченные исключения
- Стандартизированные сообщения об ошибках
- Сохранение ошибок в БД для последующего анализа
- Категоризация ошибок (валидация, бизнес-логика, системные)

### Конфигурация по профилям

**Настройка в application-{profile}.properties:**
```properties
# Silent
logging.level.com.java=WARN
logging.level.org.springframework=ERROR

# Dev  
logging.level.com.java=INFO
logging.level.org.springframework=WARN

# Verbose
logging.level.com.java=DEBUG
logging.level.org.springframework=INFO
logging.level.org.hibernate.SQL=DEBUG
```

### Принципы реализации

- **Простота**: стандартные инструменты Spring Boot
- **Производительность**: минимальное влияние на скорость работы
- **Наблюдаемость**: достаточная информация для диагностики
- **Безопасность**: защита конфиденциальных данных
- **Единообразие**: одинаковый подход во всех модулях
